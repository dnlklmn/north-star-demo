## HIGHEST PRIORITY — Asuka-Bench: Formalising Underspecified User Intent (June 4, 2026)

**Source:** [arXiv:2606.05920 — "Asuka-Bench: Benchmarking Code Agents on Underspecified User Intent and Multi-Round Refinement"](https://arxiv.org/abs/2606.05920) (June 4, 2026)

**What it does:** [Asuka-Bench](https://arxiv.org/abs/2606.05920) is a benchmark of 50 web tasks with 784 evaluation criteria and 2,402 expected outcomes, designed to test code agents on tasks where the user has *not* written a complete specification upfront. Each task is resolved through a closed loop:

1. A Code Agent generates a web project from an underspecified prompt
2. A UI Agent executes test cases on the deployed site
3. A User LLM turns evaluation outcomes into natural-language feedback for the next round

The core motivation: "users seldom write a full spec at the start; many requirements only become clear once they look at an intermediate result and react to it." The benchmark measures 8 LLMs across 2 agent frameworks, with weighted Task Pass Rate varying by 38 percentage points across models — suggesting large differences in how well agents handle refinement from feedback when the initial intent is ambiguous.

**Why this matters for North Star:**

Asuka-Bench empirically quantifies what North Star's charter is designed to prevent. The paper treats underspecified user intent as an *agent evaluation challenge* — how can agents cope without a full spec? North Star treats it as a *product design challenge* — how do you ensure the spec is complete enough before iteration begins?

These are complementary framings of the same root problem. Asuka-Bench shows that without a complete spec:
- Agent behaviour diverges wildly (38 pp spread across models)
- Evaluation results become benchmark-specific rather than reflecting real-world readiness
- Multi-round refinement is hard to measure because there's no stable ground truth

North Star's charter is the artifact that provides the stable ground truth Asuka-Bench's benchmark approach lacks. The CBAR rubric (Coverage, Balance, Alignment, Rot) is precisely the mechanism for making underspecified intent explicit *before* the agent runs.

This is the most direct academic validation of the charter-first premise found across 19 research digests. Prior work ([RUBAS](https://arxiv.org/abs/2606.04051), [Los Alamos arXiv:2605.08061](https://arxiv.org/abs/2605.08061), [Microsoft Universal Verifier](https://www.microsoft.com/en-us/research/blog/universal-verifier/)) validated the rubric structure for scoring. Asuka-Bench validates the *need* to define that rubric before the agent runs at all.

**Relevance to North Star: Methodology validation (strongest found to date for charter-first premise).** The "underspecified user intent" problem is the problem North Star solves. This paper provides a peer-reviewed empirical framing of it.

## Pre-Deployment Assurance Lane — "Toward Pre-Deployment Assurance for Enterprise AI Agents" (June 2, 2026)

**Source:** [arXiv:2606.04037 — "Toward Pre-Deployment Assurance for Enterprise AI Agents: Ontology-Grounded Simulation and Trust Certification"](https://arxiv.org/abs/2606.04037) (June 2, 2026; Golden Gate University + Novartis Healthcare)

**What it proposes:** The paper identifies a "critical gap between LLM capability benchmarking and production deployment" — post-deployment monitoring and prompt-level guardrails offer limited assurance once an agent is operating. Its proposed solution has three components:

1. **Agent Operational Envelope (AOE)** — formalises the certification space: permissions, domain constraints, safety properties, governance rules, autonomy levels
2. **Ontology-to-scenario pipeline** — derives regulatory, operational, and adversarial test scenarios automatically from the AOE
3. **Trust Certificate** — machine-verifiable attestation with graduated deployment verdicts (Approved / Conditional / Rejected)

The ontology encodes the *domain model* of what the agent is permitted to do and what constraints apply, then generates test cases from it. This is conceptually close to what a charter does: defining the task envelope before testing begins.

**How it differs from North Star:**

The Operational Envelope is a developer/compliance artifact — it formalises permissions and governance rules, not business goals and user stories. The entry point is an ontology (structured formal knowledge representation), not a conversation with a PM or domain expert. It produces a certification verdict, not a labeled dataset or scorer. And its primary audience is enterprise AI governance teams, not product builders.

North Star is upstream: the charter captures *what the agent is supposed to do and for whom* — the pre-condition for constructing any Operational Envelope. An AOE can only be meaningful if someone has already defined what "good performance" looks like for the domain.

This paper is the third pre-deployment assurance framing in the space (joining [Microsoft RAMPART](https://www.microsoft.com/en-us/research/blog/rampart/) from the May 28 digest and [Galtea](https://galtea.ai/) from the June 1 digest). None of them begin from a non-technical user's articulation of goals.

**Relevance to North Star: Positioning lane / adjacent methodology.** Validates the "define before you deploy" argument at the enterprise level. Not a direct lane competitor — developer/compliance-facing, not PM-accessible.

## Research Papers

### LongJudgeBench — Rubric-Based Judges Fail on Long-Form Outputs (June 1, 2026)

**Source:** [arXiv:2606.01629 — "Benchmarking LLM-as-a-Judge for Long-Form Output Evaluation"](https://arxiv.org/abs/2606.01629) (June 1, 2026; Tsinghua University + collaborators)

**What it finds:** [LongJudgeBench](https://arxiv.org/abs/2606.01629) is a comprehensive meta-evaluation benchmark for LLM judges on long-form outputs (documents, essays, long-form QA, extended code). The key finding: "current LLM judges remain unstable across scenarios, and rubrics or references are helpful but not always sufficient."

Specifically, rubric-guided scoring improves judge reliability over unconstrained scoring — consistent with the CBAR approach — but doesn't eliminate instability at the long-form scale. Judges trained or prompted to follow rubrics still drift when:
- The rubric criteria are genuinely multidimensional (Coverage vs. Alignment are independently hard to score across long contexts)
- The reference output is long and itself ambiguous
- Scenario complexity increases (8+ judging settings tested)

**Implications for North Star:**

CBAR applies rubric-structured scoring to eval runs. LongJudgeBench is a calibration warning for tasks where outputs are long-form (full reports, extended summaries, multi-section documents). For North Star's current scope — shorter task inputs/outputs — the instability is less acute. But as charter use cases extend to longer generation tasks, judge calibration against per-dimension rubric criteria becomes non-trivial.

The paper's finding that "rubrics are helpful but not sufficient" is important nuance: it validates rubric-structured scoring (rubrics improve reliability vs. holistic judgement) while flagging that rubric design and judge calibration must be paired. This is consistent with prior work ([iRULER / Rubric-of-Rubrics](https://chi2024.acm.org/), [Snorkel's rubric calibration series](https://snorkel.ai/blog/the-science-of-rubric-design/)) showing that rubric design quality is itself a bottleneck.

**Relevance to North Star: Methodology validation (nuanced).** Validates rubric structure; adds calibration requirement. North Star's CBAR rubric design process (the charter conversation) is where that calibration happens.

### Search-Time Contamination in Deep Research Agents (June 3, 2026)

**Source:** [arXiv:2606.05241 — "Search-Time Contamination in Deep Research Agents: Measuring Performance Inflation in Public Benchmark Evaluation"](https://arxiv.org/abs/2606.05241) (June 3, 2026)

**What it finds:** Public benchmarks become fragile for deep research agents that actively search the web during inference. The paper defines three contamination types:

1. **Benchmark Metadata Leakage** — agent retrieves the benchmark name/description
2. **Question-Context Leakage** — agent retrieves question preambles
3. **Explicit Answer Leakage** — agent retrieves ground-truth answers directly

Evaluating modern deep research agents on six public benchmarks, the paper finds Search-Time Contamination (STC) can inflate performance by up to 4%, systematically overestimating true reasoning ability.

**Relevance to North Star:** [ProjectionBench](https://arxiv.org/abs/2605.30284) (May 31 digest) showed charter-first evaluation is structurally contamination-resistant because the charter is private and bespoke. STC confirms the problem from the other side: publicly known benchmarks are increasingly unreliable for web-capable agents. Private, charter-derived datasets (what North Star produces) are the contamination-resistant alternative by construction.

**Relevance to North Star: Methodology validation.** Extends the contamination resistance argument for charter-derived private datasets.

### SoCRATES — Multi-Dimensional Benchmark for Proactive LLM Mediators (June 4, 2026)

**Source:** [arXiv:2606.05563 — "SoCRATES: Towards Reliable Automated Evaluation of Proactive LLM Mediation across Domains and Socio-cognitive Variations"](https://arxiv.org/abs/2606.05563) (June 4, 2026)

**Brief summary:** Benchmark for evaluating proactive LLM mediators in realistic conflict scenarios across 8 domains and 5 socio-cognitive axes (strategic posture, party composition, history length, emotional reactivity, cultural identity). Uses topic-localised evaluation with three complementary metrics: consensus gain, intervention timeliness, intervention effectiveness. Reaches 0.82 alignment with human experts.

**Relevance to North Star: Irrelevant to positioning.** An interesting multi-dimensional eval design in a narrow domain (mediation), but no direct connection to charter-first or CBAR methodology.
