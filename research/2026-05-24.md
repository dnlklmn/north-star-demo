## Methodology — Braintrust "Six Generations of AI Agents"

**Source:** [The six generations of AI agents and how to eval them — Braintrust Blog](https://www.braintrust.dev/blog/six-generations-ai-agents) (May 21, 2026)

Braintrust published a framework tracing how agent architectures have evolved across six generations — from simple single-turn completions through tool-calling, retrieval, multi-step pipelines, durable-state agents, and multi-agent orchestration — and how each architectural shift demands a fundamentally different evaluation approach.

**Core thesis:** Architecture follows model capability; evaluation follows architecture. At each generation:

- The **unit of evaluation** expands: from output → trace → external state
- The **eval surface** must match the system's structure: "move from 'a score' to 'a menu of scores tied to your chain's structure'"
- **Every new capability introduces a failure mode invisible to the previous generation's evals**: adding memory means you now need to test context staleness; adding tool calls means you need to test tool selection; adding multi-agent handoffs means you need to test coordination failures

**Key sentence:** *"Every new capability creates a failure mode the previous generation of evals cannot see."*

**Against the goal/story grounding rubric:**
- Does this walk a non-engineer from goals → users → stories → criteria? **No.** This is a taxonomy for eval engineers. The article opens with the assumption that you already know what you're measuring.
- Non-technical UX? **No.** Addressed to engineering leads and AI platform teams.

**Relevance to North Star: Methodology validation (structural) + competitive context.**

The post is the clearest articulation yet of why "one-shot spec generation" fails for evolving systems — if the eval surface must change at every architectural generation, the spec that defines what to measure must be revisited continuously, not generated once. North Star's charter is the input to that revisitation: as the product team's goals, user stories, and risk tolerances evolve alongside the agent architecture, the charter is the artifact that keeps the eval surface grounded. 

**Positioning implication:** Braintrust's six-generations framework is implicitly an argument for a living charter process. A product team that advances from Generation 3 to Generation 5 agents will have new failure modes their existing evals cannot see — and no structured way to decide *which new dimensions to add to the charter* without a guided discovery workflow. North Star fills that gap.

## Platform Update — OpenAI Trace Evals

**Sources:** [Trace grading — OpenAI API docs](https://developers.openai.com/api/docs/guides/trace-grading); [Evaluate agent workflows — OpenAI API docs](https://developers.openai.com/api/docs/guides/agent-evals); [Build an Agent Improvement Loop with Traces, Evals, and Codex — OpenAI Cookbook](https://developers.openai.com/cookbook/examples/agents_sdk/agent_improvement_loop) (May 12, 2026)

Following the [Promptfoo acquisition](https://openai.com/index/openai-to-acquire-promptfoo/) (covered 2026-05-21), OpenAI has shipped **Trace Evals** as a first-class feature in the [OpenAI Platform's Evals suite](https://evals.openai.com/). A *trace* captures the complete end-to-end record of an agent run — model calls, tool calls, guardrail firings, handoffs, and intermediate outputs. Trace grading assigns structured scores to those traces using configurable graders, enabling regression detection at scale across many runs.

**What's new vs. prior state:**
- Pre-acquisition OpenAI Evals was prompt/completion-level only. Trace Evals extends scoring to the full agent trajectory — matching what [AWS Strands Evals](https://github.com/strands-agents/evals) (covered 2026-05-22) introduced via `ActorSimulator` and trajectory analysis.
- Graders are the structuring layer: you define what "good" looks like per trace segment, OpenAI runs the judgment at scale.
- Available to all developers on the OpenAI Platform; no separate SDK required.

**Against the goal/story grounding rubric:**
- Guided discovery from goals → users → stories → criteria? **No.** You bring the grader definitions; OpenAI executes them.
- Non-technical UX? **Partial.** Configurable via the dashboard; grader definitions still require structured criteria specification.

**Pre-prod / prod path:**
- Pre-prod evals: **Yes**
- Prod tracing: **Yes** (captures live traces)
- BYO dataset: **Yes**
- Generates synthetic data: **No**
- LLM-as-judge: **Yes** (model-graded trace scoring)
- Custom scorers: **Yes** (configurable graders)
- Goal/story grounding: **No**
- Non-technical UX: **Partial**

**Relevance to North Star: Integration target (OpenAI stack) + competitive context.**

The North Star charter defines *what dimensions to score and why*. OpenAI Trace Evals is now the execution layer for teams on the OpenAI/Agents SDK stack: North Star upstream (charter → grader definitions), Trace Evals downstream (grades every agent run at scale). The same integration story as [Langfuse Experiments CI/CD](https://langfuse.com/docs/evaluation/experiments/experiments-ci-cd) and [AWS Strands Evals](https://github.com/strands-agents/evals), now extended to the OpenAI platform. The three together form a stack: North Star defines → [Langfuse](https://langfuse.com/) / [Strands](https://github.com/strands-agents/evals) / OpenAI Trace Evals execute, depending on the infrastructure stack.

## Research Papers — LLM Judge Reliability Under Stress

### Bias and Uncertainty in LLM-as-a-Judge Estimation

**Source:** [arXiv 2605.06939 — Bias and Uncertainty in LLM-as-a-Judge Estimation](https://arxiv.org/abs/2605.06939) (May 7, 2026, James Fiedler)

Prior work proposed bias-corrected estimators to make LLM-as-judge comparisons more rigorous. This paper stress-tests those corrected estimators and finds a critical failure mode: **when calibration datasets are shared across the models being compared (a common, practical shortcut), the corrected estimate can point in the wrong direction with high apparent confidence**.

**Key findings:**

1. **Direction reversal under shared calibration:** In a real-data MMLU-Pro case study, the corrected estimate reversed sign — concluding Model A > Model B when the truth was the opposite — while reporting high statistical confidence. The correction introduced a worse error than the uncorrected raw estimate.

2. **Judge quality as the gating condition:** The failure mode is driven by *judge quality*. High-quality judges with stable calibration produce reliable corrected estimates; low-to-medium quality judges under shared calibration fail catastrophically.

3. **Diagnostics proposed:** The paper proposes two pre-flight diagnostics — *judge quality score* and *calibration instability* — that predict when corrected estimates are likely to be unreliable. They recommend always reporting these alongside corrected estimates.

**Relevance to North Star: Direct product implication for CBAR's Alignment and Rot dimensions.** CBAR uses LLM-as-judge scoring. This paper argues that the most dangerous failure mode isn't random noise — it's systematic directional error with high confidence. When North Star scores an Alignment or Rot dimension using an LLM judge, shared calibration across rubric criteria could produce this failure. **Practical implication:** North Star's judge configuration should either (a) use dimension-specific calibration sets (avoid sharing calibration across Coverage/Balance/Alignment/Rot graders), or (b) surface the judge quality and calibration stability diagnostics from this paper as a first-class signal in the charter workflow. "This dimension's judge may be unreliable — human review recommended" is a more trustworthy product experience than a silently wrong score.

### Judging the Judges: Style Bias Is the Dominant Signal

**Source:** [arXiv 2604.23178 — Judging the Judges: A Systematic Evaluation of Bias Mitigation Strategies in LLM-as-a-Judge Pipelines](https://arxiv.org/abs/2604.23178) (April 25, 2026, Sadman Kabir Soumik)

The most comprehensive empirical comparison of LLM judge debiasing strategies to date: nine debiasing strategies across five judge models from four provider families (Google, Anthropic, OpenAI, Meta), tested on three benchmarks (MT-Bench, LLMBar, custom) against four bias types.

**Key findings:**

1. **Style bias is dominant (0.76–0.92 score) across all models tested.** This far exceeds position bias (≤ 0.04) — which has attracted most prior research attention. Style bias means judges systematically prefer responses with particular surface features (formatting, vocabulary, structural patterns) regardless of correctness.

2. **Quality-sensitive on length:** All models prefer concise responses on average, but truncation controls show they correctly distinguish quality from length (0.92–1.00 accuracy) — suggesting they reward appropriate conciseness rather than having a simple length bias.

3. **No single debiasing strategy generalises:** The nine strategies evaluated show inconsistent effectiveness across benchmarks and judge models. Practitioners should not rely on any single debiasing technique as universally reliable.

**Relevance to North Star: Calibration guidance for CBAR's judge configuration.** Style bias is the dominant failure mode in LLM evaluation, not position bias. For North Star's rubric scoring:
- **Alignment dimension** (does the output align with charter criteria?) is most exposed to style bias — a response that "sounds correct" may score well on Alignment even when it fails the underlying criterion
- **Balance dimension** (appropriate depth/coverage per criterion) is directly distorted by the same surface preferences
- **Practical implication:** Add format-neutralisation to judge prompts for Alignment and Balance scoring; test judge calibration against reformatted-but-equivalent outputs before trusting scores. [The companion paper arXiv 2605.06939](https://arxiv.org/abs/2605.06939) provides the diagnostics for detecting this before it corrupts a charter evaluation cycle.

## New Platform — Laminar

**Sources:** [Laminar — AI engineering platform for agent observability and evaluation](https://laminar.sh/); [Tech.eu — Agent debugging startup Laminar raises $3M seed to tackle the observability gap in AI agents](https://tech.eu/2026/03/17/agent-debugging-startup-laminar-raises-3m-seed-to-tackle-the-observability-gap-in-ai-agents/) (March 17, 2026); [Laminar YC company page](https://www.ycombinator.com/companies/laminar)

[Laminar](https://laminar.sh/) (YC S24) is an open-source, all-in-one platform for AI agent observability, evaluation, and debugging. It raised a $3M seed round led by Atlantic.vc in March 2026.

**Core offering:**
- **Tracing** — full observability into agent execution; structured session/trace/span hierarchy; human annotation on production traces
- **Evals** — unopinionated SDK+CLI for running evals locally or in CI/CD; integrates directly with production trace datasets so you run evals on real data, not just synthetic cases
- **Datasets** — capture and curate production traces into labeled datasets; annotation queues for human review
- **Self-hosted** — Apache 2.0 / open-source; self-hostable alternative to Langfuse and Braintrust

**Against the goal/story grounding rubric:**
- Guided discovery from goals → users → stories → criteria? **No.** Laminar is the execution and observability layer. You bring eval criteria and agent code; it instruments and scores.
- Non-technical UX? **Partial.** UI for trace review and annotation is accessible; eval definition requires SDK/code.

**Pre-prod / prod path:**
- Pre-prod evals: **Yes** (CI/CD eval runs on datasets)
- Prod tracing: **Yes** (full agent trace capture)
- BYO dataset: **Yes** (annotate and curate from production)
- Generates synthetic data: **No**
- LLM-as-judge: **Yes** (configurable)
- Prompt versioning: **Yes** (dataset versioning)
- Custom scorers: **Yes**
- Goal/story grounding: **No**
- Non-technical UX: **Partial**
- Self-hosted: **Yes**

**Relevance to North Star: Integration target (self-hosted/open-source enterprise stack).** Laminar fills the [Langfuse](https://langfuse.com/) role for teams that need on-prem or open-source infrastructure and cannot use a managed service. For enterprise buyers who want North Star's charter output to flow into an on-prem eval runner, Laminar is the third integration story alongside [Langfuse](https://langfuse.com/) (ClickHouse-managed) and [Red Hat Evaluation Hub](https://www.redhat.com/en/blog/inference-agentic-ai-scaling-enterprise-foundation-red-hat-ai-34) (Kubernetes-native). YC S24 pedigree and $3M seed give it runway to develop; worth tracking whether it adds non-technical eval-spec features.

## Methodology — evaldriven.org: EDD Codified as a Standalone Discipline

**Source:** [Eval-Driven Development — evaldriven.org](https://evaldriven.org/)

A standalone site dedicated solely to formalising Eval-Driven Development as a software engineering discipline (analogous to TDD, BDD). The site cannot be dated precisely but is indexed in May 2026 and positions itself as the canonical reference for the practice.

**Core framing:**
- *"Build evals first. Code is generated. Evals are engineered."*
- An eval is defined as: dataset + grader + harness — all three built before you write code, with the dataset evolving from synthetic to production-representative over time
- EDD's distinction from MLOps: evaluation is a *development practice that precedes writing code*, not a retrospective measurement
- Eval definitions, datasets, thresholds, and results live in version control with changelogs

**Relevance to North Star: Methodology validation + terminology convergence.** The site's "eval comes first — before the prompt, before the pipeline, before the model selection" framing is the conceptual space North Star occupies upstream of: before you can build evals, you need to know what "correct" means — which is the charter discovery problem. [evaldriven.org](https://evaldriven.org/) is the most distilled version of the EDD argument yet. No products are listed or sold; it is a methodology reference. If the site gains traction as an EDD reference hub, North Star should be positioned as the upstream specification layer the site implicitly assumes exists: *"Before your evals can come first, someone has to decide what the evals should measure. North Star is that step."*

## Positioning Hints

- **Frame the charter as an evolving artifact, not a one-time spec.** [Braintrust's six-generations framework](https://www.braintrust.dev/blog/six-generations-ai-agents) argues that every agent capability advance creates failure modes invisible to prior evals. For product teams, this means the charter needs revisiting each time the agent architecture changes. Position North Star's conversation as a *recurring check-in* rather than a pre-ship deliverable: *"As your agent gains new capabilities, your charter tells you which new dimensions to add to your eval suite — before you find out through failure."*

- **North Star now has a named downstream stack.** The integration story has three branches by infrastructure: → [Langfuse Experiments](https://langfuse.com/docs/evaluation/experiments/experiments-ci-cd) (cloud, ClickHouse-backed), → [OpenAI Trace Evals](https://developers.openai.com/api/docs/guides/trace-grading) (OpenAI stack, post-Promptfoo), → [Red Hat Evaluation Hub](https://www.redhat.com/en/blog/inference-agentic-ai-scaling-enterprise-foundation-red-hat-ai-34) / [Laminar](https://laminar.sh/) (on-prem/self-hosted). Use concretely in enterprise conversations: *"North Star produces the charter. Which runner you use — Langfuse, OpenAI Trace Evals, or Laminar — depends on your stack. We're upstream of all of them."*

- **Style bias is the argument for human-grounded criteria.** [arXiv 2604.23178](https://arxiv.org/abs/2604.23178) shows that LLM judges systematically reward surface style over correctness. A charter criterion like "the response must accurately reflect the user's stated goal" fails under a judge that rewards eloquence. Charter criteria derived from guided discovery — specific, atomic, tethered to actual user stories — are less prone to this failure than criteria auto-generated by an LLM from a vague description. *"We guide you to write criteria the judge can actually test — not criteria that sound good but reward the wrong things."*

- **Use [evaldriven.org](https://evaldriven.org/) framing as an entry point.** The site's *"Build evals first. Code is generated. Evals are engineered."* makes a strong case for pre-shipping eval specification. But it assumes you already know what the evals should measure. That's the hook: *"[Eval-Driven Development](https://evaldriven.org/) is correct — evals must come before code. But before your evals come first, you need a charter. That's what North Star produces."*

## Proposed New Matrix Columns

- **Trace-level evaluation** — Does the tool grade agent traces (full end-to-end records: model calls, tool calls, handoffs) rather than only final outputs? [OpenAI Trace Evals](https://developers.openai.com/api/docs/guides/trace-grading) and [AWS Strands Evals](https://github.com/strands-agents/evals) both do this; [Laminar](https://laminar.sh/) captures trace data for annotation and eval. This is emerging as the next standard eval surface for agentic systems.
- **Judge calibration reporting** — Does the tool surface judge quality and calibration stability diagnostics alongside eval scores? Nobody currently does this explicitly; [arXiv 2605.06939](https://arxiv.org/abs/2605.06939) argues these diagnostics are essential for trusting corrected LLM-as-judge estimates. A platform that ships this would be ahead of the research curve.
