## HIGHEST PRIORITY — Microsoft Build 2026 Eval Cluster (June 2, 2026)

### Microsoft ASSERT

**Sources:** [TechCrunch — "New Microsoft tool lets devs spin up AI behavior tests using text descriptions"](https://techcrunch.com/2026/06/02/new-microsoft-tool-lets-devs-spin-up-ai-behavior-tests-using-text-descriptions/) (June 2, 2026); [Microsoft Foundry devblog — Build 2026: Open Trust Stack for AI Agents](https://devblogs.microsoft.com/foundry/build-2026-open-trust-stack-ai-agents/)

**Adaptive Spec-driven Scoring for Evaluation and Regression Testing** — open-source framework that takes plain-language behavioral descriptions (goals, policies, "what this AI should do" and "what it must never do") and converts them into structured test cases, acceptable/unacceptable behavior definitions, scenario generation, LLM-scored evaluation, and agent execution trace logging. Cross-framework: works with LangChain, CrewAI, LiteLLM, and OpenAI agents.

Core loop:
1. Describe your AI's behavioral contract in natural language
2. ASSERT generates test scenarios and scoring criteria
3. Run against your agent
4. Log traces for regression testing

**Where it sits vs. North Star:**

- **Input layer:** ASSERT takes behavioral specs *you write yourself*. North Star extracts those specs through guided conversation — goals → users → stories → CBAR axes. ASSERT assumes you already know what to spec; North Star is upstream of that assumption.
- **Non-technical UX:** Developer-facing (CLI/SDK, requires writing policy text into a framework). No guided discovery. A PM needs an engineer to translate goals into behavioral spec language.
- **Goal/story grounding: Partial.** Accepts natural language behavioral intent as input, but no guided discovery workflow, no user/story decomposition, no structured rubric generation from conversation. It's "spec-in → tests-out," not "we help you discover the spec."

**Relevance to North Star: Adjacent lane (approaching) / Integration target.** ASSERT validates spec-first eval generation as a real pattern, but skips the discovery phase. If North Star's charter output (goals → users → stories → CBAR) is exported as a behavioral spec, ASSERT becomes a downstream runner — a potential integration target. Microsoft's distribution makes this signal important even if ASSERT doesn't enter North Star's exact lane.

### Microsoft Foundry Rubric Evaluator

**Source:** [Microsoft Foundry devblog — What's New in Microsoft Foundry at Build 2026](https://devblogs.microsoft.com/foundry/whats-new-in-microsoft-foundry-build-2026/) (June 2, 2026)

Public preview inside [Azure AI Foundry](https://azure.microsoft.com/en-us/products/ai-foundry). Two-step:
1. **Generate rubric** from the agent's definition and intended use case (LLM-assisted, no manual rubric writing)
2. **Evaluate** outputs against the generated rubric, with weighted dimensions for aggregate scoring

Feeds into **Agent Optimizer** — rubric scores drive continuous improvement and prompt tuning loops.

**Where it sits vs. North Star:**

- Auto-rubric-generation from agent context is the rubric-first eval pattern at the platform/infra layer. But the input is the agent's *technical* definition (system prompt, tools, instructions) — not a business goal / user story conversation. The rubric describes what the agent is designed to do; North Star's charter describes what the business wants outcomes to look like *for real users*.
- **Goal/story grounding: No.** Input is agent definition, not business objectives + user archetypes.
- **Non-technical UX: Partial** — cloud UI in Azure AI Foundry, but still requires building the agent first.

**Relevance to North Star: Adjacent lane / Integration target.** Foundry Rubric Evaluator sits downstream of what North Star produces. North Star's charter could be the intent input that makes Foundry Rubric Evaluator more goal-accurate — worth tracking as an integration framing.

### microsoft/eval-guide (Copilot Studio)

**Source:** [GitHub — microsoft/eval-guide](https://github.com/microsoft/eval-guide) (active; 51 commits; surfaced in Build 2026 context)

A Copilot Studio plugin implementing a four-stage guided eval lifecycle: **Discover → Plan → Generate → Interpret**. Creates single-turn and multi-turn test cases, supports seven test methods (boundary, adversarial, regression, and others), produces interactive HTML dashboards. Grounded in Microsoft's internal Eval Scenario Library and Triage Playbook. Most PM-adjacent of the three Microsoft surfaces — Copilot Studio builders include non-technical enterprise users.

**Goal/story grounding: Partial.** The Discover stage asks about intent and scenarios, but within the context of an already-built Copilot Studio agent. No goals → users → stories → charter discovery before building.

**Relevance to North Star: Adjacent lane.** Closest of the Microsoft cluster to a non-technical-user workflow, but scoped to Copilot Studio (Microsoft moat) and starts post-build, not pre-build.

### Promptfoo v0.121.14 (June 2, 2026)

**Source:** [Promptfoo GitHub releases — v0.121.14](https://github.com/promptfoo/promptfoo/releases)

New `agent-rubric` grader assertion type: unlike `llm-rubric` (which scores text output), `agent-rubric` uses a coding-agent grader that can inspect the configured workspace, tool call traces, and file system artifacts before scoring against rubric criteria. Also: seeded random sampling in CLI, trace linkage on result rows.

The `agent-rubric` grader extends rubric-first scoring from output text into agent execution traces — verifying the right tools were called, with the right evidence, in the right order. This is rubric-first scoring applied to *behavior*, not just output.

**Relevance to North Star: Integration target.** A North Star charter (CBAR axes) could supply the rubric criteria that Promptfoo's `agent-rubric` grader then evaluates against in execution.

## Research Papers

### LongJudgeBench (arXiv:2606.01629)

**Source:** [arXiv:2606.01629 — LongJudgeBench: Evaluating LLM-as-Judge on Long-Form Outputs](https://arxiv.org/abs/2606.01629) (submitted June 1, 2026)

Benchmark for LLM-as-judge systems on long-form outputs (avg. 9,000+ tokens across five real-world scenarios, six datasets, six judging protocols). Key finding: current LLM judges remain unstable on long-form outputs; rubrics and references help but are insufficient on their own.

Prior judge benchmarks focused almost exclusively on short responses — this exposes a measurement gap as agents produce longer, multi-step outputs.

**Relevance to North Star: Methodology validation.** Judge reliability degrades on the outputs charter-grounded evals most often target (long, nuanced agent responses). Charter specificity and rubric precision become *more* important at long-form length, not less.

### Agreement Metrics for LLM-as-Judge Evaluation: What to Report and Why (arXiv:2606.00093)

**Source:** [arXiv:2606.00093 — Agreement Metrics for LLM-as-Judge Evaluation: What to Report and Why](https://arxiv.org/abs/2606.00093) (Rao & Callison-Burch, UPenn; submitted May 25, 2026; not previously surfaced in this digest)

Survey of 24 LLM-as-judge papers: for binary rubric criteria (MET/UNMET), Pearson r, Spearman ρ, Kendall τ, phi coefficient, and MCC all reduce to the *same number* — reporting multiple creates false corroboration. **Cohen's κ is the one metric that adds non-redundant information** (accounts for chance agreement). Most papers over-report metrics to appear rigorous while providing no additional evidential value.

Practical implication: when validating North Star's CBAR scorers using binary MET/UNMET rubric criteria, report Cohen's κ as the primary metric.

**Relevance to North Star: Methodology validation.** Directly applicable to CBAR scorer calibration and credibility positioning.

---

### AgentCL: Toward Rigorous Evaluation of Continual Learning in Language Agents (arXiv:2606.02461)

**Source:** [arXiv:2606.02461 — AgentCL: Toward Rigorous Evaluation of Continual Learning in Language Agents](https://arxiv.org/abs/2606.02461) (submitted June 2, 2026)

Introduces a benchmark (controlled and naive task streams) and MemProbe diagnostic for evaluating how language agents accumulate and reuse experience across tasks. Tests coding, deep research, and language reasoning. Naive streams inadequate for distinguishing memory designs; controlled streams reveal plasticity-stability tradeoffs.

**Relevance to North Star: Methodology validation (secondary).** As agents gain persistent memory, charter-grounded evals need to cover consistency-over-time and goal-drift — dimensions not in current CBAR. [AgentCL](https://arxiv.org/abs/2606.02461) provides an empirical frame for a future CBAR extension.

### Mitigating Perceptual Judgment Bias in Multimodal LLM-as-a-Judge (arXiv:2606.02578)

**Source:** [arXiv:2606.02578 — Mitigating Perceptual Judgment Bias in Multimodal LLM-as-a-Judge](https://arxiv.org/abs/2606.02578) (submitted June 1, 2026)

Multimodal LLM judges reward plausible narratives over perceptually correct answers when visual evidence conflicts with text. Proposes perceptual perturbation + reward modeling as a mitigation.

**Relevance to North Star: Peripheral (methodology).** Matters if/when North Star extends to multimodal AI feature evaluation.

## Positioning Hints

- **ASSERT validates spec-first and exposes the discovery gap.** [ASSERT](https://techcrunch.com/2026/06/02/new-microsoft-tool-lets-devs-spin-up-ai-behavior-tests-using-text-descriptions/) automates spec→test conversion but assumes you have the spec. North Star's shareable framing: *"ASSERT needs a behavioral spec to work. North Star is how you write that spec — with your PM, before you touch code."*

- **Microsoft has five eval surfaces; none start before the agent is built.** The enterprise bundling play ([ASSERT](https://techcrunch.com/2026/06/02/new-microsoft-tool-lets-devs-spin-up-ai-behavior-tests-using-text-descriptions/) + [Foundry Rubric Evaluator](https://devblogs.microsoft.com/foundry/whats-new-in-microsoft-foundry-build-2026/) + [eval-guide](https://github.com/microsoft/eval-guide) + M365 CLI + Universal Verifier) creates a Microsoft eval suite. North Star's counter: *"Microsoft helps you evaluate what you shipped. North Star defines what you should have built."*

- **Foundry Rubric Evaluator as integration frame.** [Foundry Rubric Evaluator](https://devblogs.microsoft.com/foundry/whats-new-in-microsoft-foundry-build-2026/) generates rubrics from agent context; North Star generates rubrics from business goals. These are complementary: a North Star charter could be the intent input that makes Foundry's auto-rubric more accurate and goal-aligned. Worth exploring as a "bring your charter into Azure" integration story.

- **Cohen's κ for CBAR scorer credibility.** When publishing CBAR scorer validation, [arXiv:2606.00093](https://arxiv.org/abs/2606.00093) justifies reporting Cohen's κ alone for binary rubric criteria — not the full battery of redundant metrics. Cleaner and more rigorous.
