# Generated scorers

Each subdirectory here is a **scope** — a stable identifier that namespaces the scorers produced by a single prompt-eval target (or skill-eval session). Files are written by `_export_generated_scorers` in `backend/app/main.py` whenever `/sessions/{id}/generate-scorers` runs.

## Layout

```
generated/
├── <prompt_target>/             # prompt-eval scope (e.g. skill_seed/)
│   ├── <scorer_name>.py
│   └── ...
└── skill__<session_id_prefix>/  # skill-eval scope
    ├── <scorer_name>.py
    └── ...
```

## Why these are versioned

The DB stores generated scorers on each session's state — that's the operational source of truth. These files are the **versioned** mirror so:

- A regenerated scorer shows up as a diff in pull requests, not a silent DB-row swap.
- A future CI step can `braintrust push` the scorer prompts to Braintrust as online-scorer rubrics.
- The scorer rubric and the prompt that produced it can be reviewed together as code.

## Stability rules

- Re-running `/generate-scorers` on the **same session** overwrites files in place — that's why scope uses `prompt_target` for prompt-eval (a target's scorers are a single canonical set).
- Re-running on a **different session for the same skill** writes to a sibling scope directory; reconcile manually if you only want one canonical set per skill.
- Filenames are slugged (`isalnum() | -_` only, capped at 60 chars) so an LLM-emitted scorer name with whitespace or punctuation can't break the filesystem.

## What's in each file

```python
"""<description>

Generated by North Star.
session_id: <uuid>
scope: <slug>
type: alignment | coverage | safety
generated_at: <iso8601>
"""

def <scorer_name>(output: str, input: str) -> float:
    ...  # the LLM-as-judge function as the prompt builder produced it
```

The body is whatever `call_generate_scorers` emitted; it's expected to call a `call_judge(prompt)` helper that the eval runner provides at execution time. Don't import these at module-load — they have no `import` statements and rely on a runtime-injected helper.

## Cleanup

Stale scopes from deleted prompt-eval projects don't auto-clean. If a session is deleted, its generated scorer files stay on disk; a periodic pruning script (or manual `git rm`) handles that.
